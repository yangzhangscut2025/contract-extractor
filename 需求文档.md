## 1. 项目背景与目标

### 1.1 背景

**公司需要处理来自不同国家、格式不统一的劳动合同及调薪文件（PDF格式），人工录入关键字段效率低、易出错。现有文件由系统导出，命名有规律，但文件内部** **不包含合同编号** **，需要工具自动生成合同编号来区分同一员工的多个合同。文件类型多样（文字型PDF、扫描件、多语言混合）。**

### 1.2 目标

**开发一个桌面辅助工具（基于Electron），实现：**

* **批量导入 PDF 文件**
* **自动识别文件类型（劳动合同 / 调薪文件 / 其他）**
* **自动提取 42 个关键字段（劳动合同）+ 3 个系统生成字段 = ****45 列**
* **自动生成合同编号（同一员工按顺序递增）**
* **对于文字型 PDF 直接提取文本，对于扫描件/图片型 PDF 调用阿里云 OCR**
* **人工校对与修正**
* **导出为标准 Excel 模板（45 列）**

### 1.3 适用范围

* **输入** **：系统导出的 PDF 文件，文件命名格式固定（见 2.1），文件内容不包含合同编号**
* **处理对象** **：劳动合同、调薪文件（其他类型文件自动跳过或标记）**
* **语言支持** **：英文、泰文、中文等多语言混合，输出优先英文，若无英文则保留原文**

## 2. 业务流程

### 2.1 文件命名规则（已知条件）

* **文件名示例：**`<span class="ne-text">CE3E5PLL__1.Mr.Chanon - Employment_agreement Thai-English_20240228121302A920.pdf</span>`
* **解析规则：**

  * **取****第一个** `<span class="ne-text">__</span>`（双下划线）**左侧**部分作为员工系统编号（如 `<span class="ne-text">CE3E5PLL</span>`）
  * **若文件名中不包含 **`<span class="ne-text">__</span>`，则取前 8 个字符作为员工编号（此情况实际不会发生，但保留容错）
* **员工编号长度可能不等于 8，直接使用原字符串（不截断、不补位）**

### 2.2 整体处理流程

```plain
用户选择文件（单个或批量）
    ↓
解析文件名 → 提取员工系统编号（第一个__左侧）
    ↓
读取文件内容
    ↓
┌─────────────────────────────────────┐
│ 第一步：文件类型识别                  │
│ - 先尝试从文件内容关键词判断           │
│ - 若不确定，调用大模型快速分类         │
│ - 结果：劳动合同 / 调薪文件 / 其他     │
└─────────────────────────────────────┘
    ↓
若为“其他” → 跳过，记录日志，提醒用户
    ↓
若为“劳动合同”或“调薪文件” → 继续
    ↓
┌─────────────────────────────────────┐
│ 第二步：文本提取                      │
│ 1. 尝试用 pdf.js 提取文字层           │
│ 2. 若失败（空白或乱码）→ 转为图片 →   │
│    调用阿里云 OCR API                  │
│ 3. 若 OCR 仍失败 → 标记“不可识别”，    │
│    提醒人工审核（用户自行处理，不自动重试）│
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 第三步：合同编号自动生成               │
│ - 格式：员工系统编号 + "_" + 序号     │
│ - 序号根据历史记录中同一员工的最大序号+1 │
│   （第一份为 _01，第二份 _02）        │
│ - 若用户删除某记录后重新导入，序号仍取 │
│   当前历史中的最大序号+1（不填补空缺）  │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 第四步：确定合同类别                   │
│ - 取自第一步的文件类型识别结果         │
│ - 取值：劳动合同 / 调薪文件            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 第五步：信息提取（大模型 API）          │
│ - 劳动合同和调薪文件统一使用 42 字段 Prompt │
│   （见附录A），LLM 从原文中提取存在的字段，  │
│   缺失字段值为 null                          │
│ - 调用 DeepSeek 或 智谱 API            │
│ - 输出 JSON（缺失字段值为 null）        │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 第六步：后处理校验                     │
│ - 邮箱格式、日期格式、电话格式验证     │
│ - 金额数字格式验证                     │
│ - 不符合规则的字段增加“待核对”标记     │
└─────────────────────────────────────┘
    ↓
展示在 UI 上（左侧原文，右侧提取结果表格）
    ↓
人工校对、修改字段值
    ↓
导出为 Excel（45 列，空值显示为空白单元格）
```

## 3. 功能需求详述

### 3.1 文件输入与管理

| **功能点**        | **描述**                                                                                  |
| ----------------------- | ----------------------------------------------------------------------------------------------- |
| **单个/批量导入** | **支持选择多个 PDF 文件，显示进度条**                                                     |
| **文件列表**      | **展示已导入文件名、员工编号、合同编号、合同类别、处理状态（待处理/处理中/已完成/失败）** |
| **拖拽上传**      | **支持拖拽 PDF 文件到窗口**                                                               |
| **历史记录**      | **自动保存处理结果到本地数据库（SQLite），支持再次打开查看**                              |

### 3.2 文件类型识别（用于确定“合同类别”）

* **规则判断** **：扫描文件内容（前 2000 字符）是否包含关键词 **`<span class="ne-text">employment agreement</span>`, `<span class="ne-text">employment contract</span>`, `<span class="ne-text">salary adjustment</span>`, `<span class="ne-text">salary increase</span>` 等。匹配到劳动合同类关键词则分类为 **劳动合同** **，匹配到调薪类关键词则分类为** **调薪文件** **。**
* **大模型快速分类** **：若规则无法确定，调用大模型，Prompt 如下：**

```plain
判断以下文档属于哪一类，只输出一个单词：
- EmploymentContract
- SalaryAdjustment
- Other
文档开头部分：{...}
```

* **处理结果** **：**
* **劳动合同 → 合同类别 = “劳动合同”，走完整提取流程**
* **调薪文件 → 合同类别 = “调薪文件”，走精简提取流程**
* **其他 → 跳过，标记“不处理”，并在界面列出供用户忽略或强制处理**

### 3.3 文本提取

| **场景**              | **处理方法**                                                                                                                        |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **文字型 PDF**        | **使用** `<span class="ne-text">pdf.js</span>`的 `<span class="ne-text">getTextContent</span>`提取，保留段落结构                |
| **扫描件/图片型 PDF** | **将 PDF 每一页转为图片（Electron 主进程），调用阿里云 OCR API（**`<span class="ne-text">RecognizePdf</span>`接口），获取识别文本 |
| **混合型 PDF**        | **优先提取文字层，若某页无文字则对该页单独 OCR**                                                                                    |
| **完全失败**          | **标记“不可识别”，弹出提示框告知用户，工具不进行自动重试。用户自行处理后（如更换文件或手动粘贴）再继续。**                        |

 **OCR 配置** **：**

* **服务商：阿里云 OCR（通用文字识别高级版）**
* **API 接入：使用 **`<span class="ne-text">@alicloud/ocr-api20210707</span>` SDK
* **区域：**`<span class="ne-text">cn-hangzhou</span>`
* **并发控制：同时最多 3 个请求，避免限流**

### 3.4 合同编号自动生成

* **来源** **：** **不依赖 PDF 内容** **，完全由工具自动生成。**
* **格式** **：**`<span class="ne-text">员工系统编号 + "_" + 序号</span>`
* **员工系统编号：从文件名解析（第一个 **`<span class="ne-text">__</span>` 左侧）
* **序号：2 位数字，不足 2 位补零（**`<span class="ne-text">01</span>`, `<span class="ne-text">02</span>`, ...）
* **序号生成规则** **：**
* **查询本地历史记录中****同一员工**的最大序号，在此基础上 +1
* **若无历史记录（该员工第一份文件），序号从 **`<span class="ne-text">01</span>` 开始
* **若用户删除某条记录后重新导入该文件，序号仍取当前历史中的最大序号 +1（不填补已删除的序号空缺）**
* **示例** **：**
* **员工编号：**`<span class="ne-text">CE3E5PLL</span>`
* **已有合同：**`<span class="ne-text">CE3E5PLL_01</span>`、`<span class="ne-text">CE3E5PLL_02</span>`（假设 `<span class="ne-text">_02</span>` 被删除）
* **新合同自动生成：**`<span class="ne-text">CE3E5PLL_03</span>`（而不是 `<span class="ne-text">_02</span>`）
* **注意事项** **：**
* **如果用户同时导入多个同一员工的文件，按文件导入顺序依次分配序号（先 **`<span class="ne-text">_01</span>`，后 `<span class="ne-text">_02</span>`）。

### 3.5 合同类别

* **字段位置** **：紧跟在“合同编号”之后。**
* **取值** **：**`<span class="ne-text">劳动合同</span>` 或 `<span class="ne-text">调薪文件</span>`（取自 3.2 的文件类型识别结果）。
* **用途** **：决定后续使用哪一套 Prompt 模板和字段集合。**

### 3.6 信息提取（大模型）

#### 3.6.1 劳动合同提取字段（42 个）

**详见附录 A。字段定义包含：**

* **基础信息：法定姓名、性别、国籍、生日、年龄、婚姻状况、出身地**
* **雇佣信息：雇主名称、就职国家/地区、岗位名称、入职时间、合同类型**
* **薪酬：税前年薪/月薪及币种、交通补贴、餐补、奖金、其他补贴**
* **合同期限：合同生效/结束日期、试用期**
* **休假：年假天数、病假天数**
* **银行信息：银行名称、地址、账户名、账号、IBAN、SWIFT**
* **联系方式：个人邮箱、工作邮箱、电话号码、居住地址、身份证件、社保号**

 **注意** **：合同编号、合同类别、员工系统编号不由大模型提取，由工具自动生成。**

#### 3.6.2 调薪文件提取字段

调薪文件与劳动合同使用相同的 42 字段 Prompt（见附录 A）。调薪文件通常包含薪酬、补贴、银行信息等变更内容，LLM 自动提取原文中存在的字段，缺失字段值为 null。导出 Excel 时所有列均参与填充，无需用户手动补充。

#### 3.6.3 日期字段提取口径（重要）

* **入职时间（start_date）** **：**
* **优先提取合同中明确标注为 **`<span class="ne-text">Start Date</span>`, `<span class="ne-text">Commencement Date</span>`, `<span class="ne-text">Date of Employment</span>`, `<span class="ne-text">入职日期</span>` 等字段的值。
* **若没有明确标注，则** **默认为合同生效日期** **（如果有）。**
* **若两者都没有，则为 **`<span class="ne-text">null</span>`。
* **合同生效日期（contract_start_date）** **：**
* **优先提取合同中明确标注为 **`<span class="ne-text">Effective Date</span>`, `<span class="ne-text">Contract Effective Date</span>`, `<span class="ne-text">本合同生效日期</span>` 等字段的值。
* **若没有明确标注，则** **默认为签署日期（Signature Date）** **。**
* **若都没有，则为 **`<span class="ne-text">null</span>`。

#### 3.6.4 大模型调用要求

* **API** **：DeepSeek 或 智谱 AI（用户可配置）**
* **Prompt 模板** **：见附录 B（劳动合同）和附录 C（调薪文件）**
* **输出格式** **：纯 JSON，缺失字段输出 **`<span class="ne-text">null</span>`（不加引号，如 `<span class="ne-text">"annual_gross_salary": null</span>`）
* **重试机制** **：单次调用失败重试 2 次，间隔 2 秒**
* **并发控制** **：串行处理每个文件，避免超出 token 限制**

### 3.7 后处理校验

**对每个提取出的字段值进行格式验证，不符合规则的字段自动标记，展示时高亮提示。**

| **字段类型** | **校验规则**                                                                                                                                                              | **标记内容**                              |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| **邮箱**     | **正则** `<span class="ne-text">^[^\s@]+@([^\s@]+\.)+[^\s@]+$</span>`                                                                                                   | `<span class="ne-text">"待核对邮箱"</span>`   |
| **日期**     | **能成功解析** `<span class="ne-text">YYYY-MM-DD</span>`，且范围合理（1900-2100）                                                                                       | `<span class="ne-text">"日期格式异常"</span>` |
| **电话号码** | **去除空格、**`<span class="ne-text">+</span>`、`<span class="ne-text">-</span>`、`<span class="ne-text">(</span>``<span class="ne-text">)</span>`后，数字长度 ≥ 8 | `<span class="ne-text">"电话格式可疑"</span>` |
| **金额数字** | **只允许数字和一个小数点，不能有字母或货币符号**                                                                                                                          | `<span class="ne-text">"金额待清理"</span>`   |
| **年龄**     | **0-120 之间的整数**                                                                                                                                                      | `<span class="ne-text">"年龄超出范围"</span>` |

**校验结果不修改原始提取值，只在 UI 上增加警告图标，并在导出时增加一列“校验状态”。**

### 3.8 人工校对界面

* **左右布局** **：左侧展示原始合同文本（或 OCR 识别文本），右侧展示提取出的字段表格（可编辑）。**
* **表格功能** **：**
* **每行显示：字段名、提取值、校验状态、人工修正值**
* **支持直接修改单元格内容**
* **修改后自动保存到本地**
* **高亮定位** **：点击任意字段值，左侧文本区域自动滚动并高亮显示该关键词出现的位置（模糊匹配）。**
* **批量操作** **：支持标记“已验证”、批量导出。**

### 3.9 导出功能

* **格式** **：Excel (.xlsx)**
* **模板** **：45 列（3 列系统生成 + 40 列大模型提取），调薪文件只填充其中 6 列（员工编号、合同编号、合同类别、full_name、effective_date 等），其余留空。**
* **空值处理** **：大模型输出 **`<span class="ne-text">null</span>` 时，导出为 **空白单元格** **（不显示 "null" 字符串）；数字类型字段若为 null，也是空白单元格（不是 0）。**
* **导出范围** **：当前选中的文件或全部已处理文件。**
* **文件命名** **：**`<span class="ne-text">合同提取结果_年月日时分秒.xlsx</span>`

### 3.10 配置管理

* **API 选择** **：DeepSeek / 智谱 AI**
* **API Key** **：加密存储到本地（**`<span class="ne-text">electron-store</span>`）
* **模型名称** **：可自定义（如 **`<span class="ne-text">deepseek-chat</span>`, `<span class="ne-text">glm-4</span>`）
* **OCR AccessKey/Secret** **：存储在本地，可测试连接**
* **输出语言偏好** **：默认“优先英文，无则原文”**

## 4. 异常处理与边界情况

| **异常场景**                                       | **处理方式**                                         |
| -------------------------------------------------------- | ---------------------------------------------------------- |
| **文件非 PDF 格式**                                | **提示不支持，跳过**                                 |
| **PDF 已加密**                                     | **尝试空密码解锁，失败则要求用户提供密码**           |
| **网络中断**                                       | **自动暂停，恢复后续传**                             |
| **大模型 API 返回非 JSON**                         | **重试一次，仍失败则显示原始输出，允许用户手动纠正** |
| **OCR 额度不足**                                   | **提示用户购买资源包或切换账号**                     |
| **文件名中无** `<span class="ne-text">__</span>` | **回退到取前 8 字符（实际不会发生）**                |
| **同一文件重复导入**                               | **根据文件 MD5 去重，提示“已处理过”**              |
| **历史记录中同一员工最大序号异常**                 | **强制从 01 开始，并记录警告日志**                   |
| **调薪文件导出时未提取字段**                       | **使用 42 字段全量提取，自动填充**                             |

## 5. 非功能需求

### 5.1 性能要求

* **单个文件从导入到展示提取结果（含 OCR+大模型）平均 ≤ 20 秒**
* **批量处理 10 个文件时，界面不卡顿，支持取消操作**
* **OCR 并发限制 3 个，大模型串行**

### 5.2 数据安全

* **所有 API Key、OCR 凭证仅存储在本地，不上传任何服务器**
* **合同文件内容不离开用户电脑（除调用 API 外，不保存到第三方）**
* **日志中不记录敏感字段（如身份证号、银行账号）**

### 5.3 兼容性

* **操作系统：Windows 10/11, macOS 11+**
* **屏幕分辨率：最小 1366×768，推荐 1920×1080**
* **无网络时仍可打开工具查看历史记录，但无法进行新文件处理**

### 5.4 可维护性

* **代码模块化：文件解析、OCR、大模型调用、校验、UI 相互解耦**
* **字段配置化：字段列表、校验规则可热更新（JSON 配置文件）**
* **日志记录：所有 API 请求响应、错误信息保存到本地文件，便于排查**
